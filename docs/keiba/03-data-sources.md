# データソースと JRDB 仕様リファレンス

## 1. データソース選定結果

### 1.1 採用: JRDB ベーシック

| 項目 | 詳細 |
|------|------|
| **プラン** | ベーシック |
| **費用** | 月額 2,380 円（初回 3,000 円）。2025 年 4 月改定後の価格 |
| **データ範囲** | 1999 年〜現在（本プロジェクトでは 2015 年以降を使用） |
| **インポート範囲** | 2015 年〜現在（約 10 年分、約 35,000 レース） |
| **形式** | 固定長テキスト（CP932 エンコーディング）、lzh/zip 圧縮 |
| **法的リスク** | なし（正規サービス） |

### 1.2 プラン選定理由（ベーシック vs アドバンス）

| 要素 | ベーシック（2,380 円/月） | アドバンス（2,880 円/月） |
|------|------------------------|------------------------|
| 各種競馬新聞 | ○ | ○ |
| 競馬予想データ（プログラム用） | ○ | ○ |
| スタッフ予想コラム | ○ | ○ |
| コメント直前情報（パドック・返し馬） | × | ○ |

- **プログラム用データファイル（KYI, SED, BAC, HJC, TYB 等）はベーシックで全て取得可能**
- アドバンスの追加要素「コメント直前情報」は人間が書いたテキストコメントであり、構造化された TYB データファイルとは別物
- NLP ベースのパドック分析を検討する段階（Phase 5 以降）まで不要
- **結論: ベーシックで十分**

### 1.3 JRDB 採用の決め手

1. **法的安全性**: 正規の有料サービスで利用規約上の問題なし
2. **安定性**: 固定長フォーマットは変更歴がなく、一度パーサーを書けば半永久的に使える
3. **データの質**: IDM・調教指数等は他のデータソースでは得られない差別化要因
4. **コスト**: 月額 2,380 円は投資対効果として十分妥当

### 1.4 不採用候補とその理由

| ソース | 不採用理由 |
|--------|-----------|
| netkeiba スクレイピング | 利用規約で明確にスクレイピング禁止。IP ブロックあり。法的リスク高 |
| JRA-VAN Data Lab | Windows 前提の COM API。macOS 非対応で Python 連携が困難 |
| JRA 公式サイト | doAction/POST 方式で実装が煩雑。血統・過去成績の詳細なし |

### 1.5 データ範囲の選定理由（2015 年以降）

- 学習には 5-10 年分のデータで十分
- 古いデータは競馬場改修（京都 2020-2023 等）で現在の傾向と乖離するリスクがある
- インポート時間が短く、イテレーションが速い
- 必要に応じて後から 1999 年まで拡張可能

### 1.6 ダウンローダーの方針

- GitHub の [JRDB-data/downloader](https://github.com/JRDB-data/downloader) は最終更新 2023 年 8 月で事実上メンテナンスされていない
- **自前でダウンローダーとパーサーを実装する**（`keiba/scraper/jrdb.py`）
- JRDB 会員ページからの HTTP ダウンロード + lzh/zip 解凍 + 固定長パースの一連のパイプラインを構築

## 3. ファイル形式の共通仕様

### 3.1 エンコーディングとフォーマット

- **文字コード**: CP932（Windows 拡張 Shift-JIS）
- **圧縮**: lzh 形式または zip 形式
- **データ形式**: 固定長テキスト。各フィールドはバイト位置と長さで指定
- **改行**: CRLF

### 3.2 ファイル名構成

```
データ識別名 + 版数 + 年月日(yyMMdd)
例: KYH150105.lzh  →  KY(競走馬データ) + H(版数) + 150105(2015年1月5日)
```

### 3.3 マスタデータの更新ルール

- 騎手・調教師の勝数などの成績は **週単位** で更新
- **土曜日データ**: その週の前の木曜日に更新（その週が始まる前の成績）
- **日曜日データ**: 月曜日に更新（その週の結果を踏まえた成績）
- **重要**: 土曜日の成績は日曜日のデータに反映されない

## 4. 連携キー構造

JRDB データ間のリレーションは以下のキーで連携:

| キー | 用途 | 含まれるファイル |
|------|------|----------------|
| レースキー | レースを一意に特定 | KYI, SED, BAC, HJC |
| レースキー + 馬番 | 出走馬を一意に特定 | KYI（馬データ）, TYB（直前情報） |
| 血統登録番号 | 馬を一意に特定 | KYI, 馬基本データ |
| 競走成績キー | レース結果を一意に特定 | SED |
| 騎手コード | 騎手を一意に特定 | KYI, KSA/KZA |
| 調教師コード | 調教師を一意に特定 | KYI, CSA/CZA |

### 4.1 データ連携の流れ

```
BAC（番組データ） ─── レースキー ──→ KYI（競走馬データ）
                                        │
                    ┌── 血統登録番号 ──→ 馬基本データ
                    ├── 騎手コード   ──→ KSA（騎手マスタ）
                    ├── 調教師コード ──→ CSA（調教師マスタ）
                    └── 前走成績キー ──→ SED（成績データ）
```

## 5. 固定長パーサー（実装済み）

実装: `src/providence/keiba/scraper/parser.py`

### 5.1 設計

```python
@dataclass(frozen=True)
class FieldSpec:
    name: str
    start: int       # JRDB 仕様の「開始」列（1始まり）
    length: int      # バイト長
    dtype: str        # "str" | "int" | "float" | "hex"
    scale: float = 1.0  # 乗算スケール。raw 値に scale を掛ける
```

- `parse_file(data, specs, record_length)` でファイル全体を一括パース
- CRLF で行分割し、各行のバイト長を `record_length` と照合
- 不一致行は非 strict モードでスキップ（ログ出力）

### 5.2 実装で確認済みの注意事項

- **レコード長**: 仕様書の「レコード長」は CRLF を含む。パーサーは CRLF split 後の行長で判定するため、**実際のレコード長 = 仕様書 - 2 bytes**

| ファイル | 仕様書の値 | 実際のレコード長（CRLF 除外） |
|---------|-----------|---------------------------|
| BAC | 176 (+CRLF=178, +WIN5等=184) | **182** |
| KYI | 1024 | **1022** |
| SED | 376 | **374** |
| HJC | 444 | **442** |
| UKC | 292 | **290** |
| KSA/CSA | 272 | **270** |

- **フィールド位置**: 仕様書は版数によりフィールドが追加される。特に KYI は第 11 版まで拡張されており、騎手コード（第 5 版追加、start=336）と調教師コード（start=341）の位置に注意
- **IDM 等の指数**: KYI の `ZZ9.9` 形式は既に小数付きで格納。scale は **1.0**（0.1 を掛けてはならない）
- **SED の race_time**: `9999` 形式で 1byte 目が分、2-4byte 目が秒×10。例: `1557` = 1 分 55.7 秒
- **SED アーカイブに SRB が混在**: SRB は拡張版でレコード長が異なる（850 bytes）。パーサーのレコード長チェックで自動スキップ
- **lhafile**: Python 3.12+ で動作確認済み

## 7. ダウンロード制限と運用上の注意

### 7.1 レース開催日の過去データ制限

JRDB は**レース開催日（土日祝）の特定時間帯に過去データの年間一括ダウンロードを制限**する。

- **403 レスポンス**: 「只今の時間帯の過去ＪＲＤＢデータへのアクセスは制限されています。」
- **日次ファイル（当日分）**: いつでも OK
- **年間一括ファイル（過去データ）**: レース開催日は制限。**平日に実行すること**
- 参照: http://www.jrdb.com/member/bbs3/c-board.cgi?cmd=one;no=38;id= （会員限定）

### 7.2 年間一括と日次ファイルの関係

年間一括ファイル（例: `SED_2024.lzh`）は、その年の日次ファイル（`SED240106.txt` 等）を全てまとめた圧縮アーカイブ。**データの中身は完全に同一**。年間一括は 1 リクエストで 1 年分を取得できるため効率的。

### 7.3 ダウンロード URL パターン

| パターン | URL 構造 | 認証 |
|---------|---------|------|
| 日次 | `http://www.jrdb.com/member/data/{Dir}/{PREFIX}{yyMMdd}.lzh` | HTTP Basic Auth |
| 年間一括 | `http://www.jrdb.com/member/data/{Dir}/{PREFIX}_{year}.lzh` | HTTP Basic Auth |

Dir とプレフィックスの対応:

| Dir | 日次プレフィックス | 年間プレフィックス |
|-----|------------------|------------------|
| Ks | KSA | KSA |
| Cs | CSA | CSA |
| Ukc | UKC | UKC |
| Bac | BAC | BAC |
| Kyi | KYI | KYI |
| Sed | SED | SED |
| Hjc | HJC | HJC |

## 8. リスクと緩和策

| リスク | 影響 | 緩和策 |
|--------|------|--------|
| JRDB サービス廃止（個人運営） | データ取得不能 | パーサーを抽象化し別ソースに切替可能にする。取得済みデータはローカル DB に永続化 |
| レース開催日のダウンロード制限 | バルクインポート不可 | 平日に年間一括を実行。間隔を 10 秒以上に設定 |
| フォーマットの経年変化 | パースエラー | パーサーの非 strict モードで不一致行をスキップ。ログで検知 |
| lzh 圧縮の Python 対応 | — | `lhafile>=0.3` で Python 3.12+ 動作確認済み |
