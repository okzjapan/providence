# 陥りやすい罠と対策チェックリスト

競馬 AI 開発で陥りやすい問題をチェックリスト形式で整理する。各項目に「検知方法」と「対策」を明記。

---

## 1. データリーケージ（時系列リーク）

**問題**: 予測時点で利用できない情報が訓練データまたは特徴量に混入し、バックテストが実運用より不当に良い結果を示す。

### チェックリスト

- [ ] 確定着順をラベル以外に使用していないか
- [ ] 当日確定オッズを前日予測モデルに含めていないか
- [ ] 馬体重を前日予測モデルに含めていないか
- [ ] 返し馬・パドック評価を前日予測モデルに含めていないか
- [ ] 特徴量計算で「未来の成績」を集計に含めていないか
- [ ] 訓練 / 検証 / テスト分割が時系列順になっているか
- [ ] 騎手・調教師の通算成績に「未来のレース」が含まれていないか

**検知方法**:
- `assert_no_leakage` テストを実装し、特徴量 DataFrame にリーク候補カラムが含まれていないことを検証
- バックテストの回収率が不自然に高い（150% 超など）場合はリークを疑う

**対策**:
- 特徴量計算時に必ず `as_of_date` パラメータを使い、カットオフ日以降のデータを除外
- 情報公開タイムラインを `01-jra-structure.md` セクション 13 で管理し、各特徴量の利用可能時点を明示

---

## 2. 過学習（Overfitting）

**問題**: 訓練データに過度にフィットし、未知データでの性能が低下する。

### チェックリスト

- [ ] Early Stopping を設定しているか（`early_stopping_rounds >= 50`）
- [ ] 時系列分割を使っているか（ランダム分割は不可）
- [ ] 訓練セットと検証セットの NDCG 乖離が大きくないか
- [ ] セグメント分割後の各セグメントに十分なデータ量があるか
- [ ] 特徴量数がデータ量に対して過剰でないか

**検知方法**:
- 訓練 NDCG vs 検証 NDCG のギャップを監視。0.05 以上の乖離は過学習のサイン
- 複数の時系列ウィンドウでの回収率の分散を確認

**対策**:
- `num_leaves` を 31 以下に制限（浅い木でアンダーフィット気味にする）
- `min_data_in_leaf` を 20 以上に設定
- `feature_fraction` と `bagging_fraction` を 0.8 程度に設定（ランダム性の注入）
- 特徴量選択で 200+ 次元 → 50-80 次元に絞り込み

## 3. 的中率と回収率の混同

**問題**: 「予測を当てること」に最適化すると、人気馬ばかり推奨して控除率に負ける。

### チェックリスト

- [ ] 評価指標に回収率シミュレーションを含めているか
- [ ] NDCG だけでなく期待値ベースの検証を行っているか
- [ ] 人気上位馬への偏りが過剰でないか確認しているか

**検知方法**:
- 的中率 30% 超でも回収率 < 80% の場合、人気馬偏重のサイン
- 推奨馬の平均オッズが 3 倍未満なら期待値が取りにくい

**対策**:
- **目的は「着順を当てる」ではなく「市場が過小評価している馬を見つける」**
- 期待値 > 1.2 の馬券のみを推奨対象にするフィルタを適用
- 回収率を最終的な判断基準にし、NDCG は補助指標として使う

## 4. データ品質

**問題**: 出走取消・失格・競走中止の不適切な処理、経年のフォーマット変化。

### チェックリスト

- [ ] 出走取消の馬を特徴量計算から除外しているか
- [ ] 失格・降着後の確定着順を使用しているか
- [ ] 競走中止の馬を着順なし（欠損）として処理しているか
- [ ] ローテーション計算で出走取消を考慮しているか
- [ ] JRDB 経年データで年度ごとの整合性を確認したか

**検知方法**:
- 年度ごとのレコード数・欠損率・カラム分布を集計して異常を検出
- 特定年度で極端にパフォーマンスが変わる場合はデータ品質を疑う

**対策**:
- 初回インポート時に年度ごとのデータ検証スクリプトを実行
- 出走取消・失格・競走中止の処理は `entry_status` フィールドで一元管理
- 京都競馬場改修（2020-2023）等の構造変化を考慮し、基準タイムの算出を分離

## 5. Plackett-Luce 計算量

**問題**: 18 頭の 3 連単は 4,896 通りあり、オートレースの 15 倍の計算量。

### チェックリスト

- [ ] 全券種の確率計算がタイムアウトしないか確認しているか
- [ ] 少頭数レース（5 頭等）での動作を確認しているか

**検知方法**:
- 1 レースあたりの確率計算時間を計測。1 秒を超えたら最適化を検討

**対策**:
- 初期は全組み合わせ列挙で OK
- ボトルネックが発生したらスコア上位 N 頭（例: 上位 10 頭）に限定して近似
- NumPy のベクトル化計算で高速化

---

## 6. 市場エッジの縮小

**問題**: AI 競馬参入者の増加により、市場効率性が年々改善し、エッジが縮小する。

### チェックリスト

- [ ] 定期的（四半期ごと等）に回収率の推移をモニタリングしているか
- [ ] エッジの源泉が「一般購入者が使いにくいデータ」に依拠しているか

**検知方法**:
- 四半期ごとの回収率トレンド分析
- モデルの feature importance の変動監視

**対策**:
- JRDB 独自指数（IDM・調教指数等）など、一般購入者がアクセスしにくいデータを差別化の軸にする
- 調教データ・パドック情報など「手間がかかるデータ」の活用
- モデルの定期的な再学習とフィードバックループの構築

---

## 7. JRDB サービスの継続性

**問題**: JRDB は個人運営のため、サービス廃止のリスクがある。

### チェックリスト

- [ ] データ取得層が抽象化されており、別データソースに差し替え可能か

**対策**:
- パーサーを `DataSourceInterface` のような抽象層を通じて実装
- JRDB 廃止時は JRA-VAN（Windows VM 経由）や別サービスに切り替え可能にしておく
- 取得済みデータはローカル DB に永続化しておく（JRDB が止まっても過去データは手元にある）

---

## 8. 競馬場改修の影響

**問題**: 競馬場の改修により、改修前のデータが現在の馬場傾向と乖離する。

### チェックリスト

- [ ] 改修前後のデータを区別しているか
- [ ] 基準タイム・馬場指数の算出が改修前後で分離されているか
- [ ] コース適性の計算が改修後データのみで行われているか

**検知方法**:
- 同一競馬場の改修前後でタイム分布を比較

**対策**:
- 改修後フラグをデータに付与
- 条件別適性は改修後データのみで計算
- 京都（2020-2023 改修）、中京（2012 改修）の履歴を管理
