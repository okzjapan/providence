# 陥りやすい罠と対策チェックリスト

オートレース AI 開発で注意すべき問題をチェックリスト形式で整理する。

---

## 1. データリーケージ

**問題**: 予測時点で利用できない情報が特徴量に混入する。

### チェックリスト

- [ ] `finish_position` をラベル以外に使用していないか
- [ ] `race_time`, `start_timing`, `accident_code` を特徴量に含めていないか
- [ ] 選手成績の集計に未来のレースを含めていないか（`d < current_date` の条件があるか）
- [ ] `load_history(as_of_date)` で同日リークを防止しているか
- [ ] `assert_no_leakage` テストが通るか

**検知方法**: バックテストの回収率が不自然に高い場合はリークを疑う。

---

## 2. 試走タイム未取得

**問題**: 試走がまだ行われていない段階で予測を実行すると、最も重要な特徴量が欠損する。

### チェックリスト

- [ ] 予測前に試走タイムが `race_entries` に反映されているか確認しているか
- [ ] CLI で試走未取得の警告が表示されていないか

**対策**: `predict.py` CLI で試走未取得車への警告を表示。Phase 1 で Program を再取得して試走タイムを反映。

---

## 3. オッズ取得タイミング

**問題**: `judgment_time` 以降のオッズを使うとリーク。バックテストで最終オッズを使うと実運用と乖離。

### チェックリスト

- [ ] `judgment_time` 以前の最新 `ingestion_batch_id` のみを使用しているか
- [ ] バックテストで `judgment_time` を正しく設定しているか

**対策**: `Repository.get_latest_market_odds` が `judgment_time` フィルタを自動適用。バックテストでは historical replay（最終オッズ）と live replay（時刻指定オッズ）を区別。

---

## 4. 出走取消・落車・失格

**問題**: 特殊な出走状態のハンドリングが不適切だとバックテスト結果が歪む。

### チェックリスト

- [ ] 出走取消の車番を特徴量計算から除外しているか
- [ ] 払い戻しに `ticket_payouts` の確定データを使っているか
- [ ] `entry_status` = 取消の場合、返還処理を反映しているか

---

## 5. 過学習

**問題**: レース数が少ない（年間約 10,000-15,000 レース）ため、過学習のリスクがある。

### チェックリスト

- [ ] Early Stopping を使用しているか
- [ ] 時系列分割を使用しているか
- [ ] 訓練 NDCG と検証 NDCG の乖離を監視しているか

**対策**: `num_leaves` の制限、`feature_fraction` と `bagging_fraction` によるランダム性の注入。

---

## 6. 複勝の未対応

**問題**: 現在の Kelly 最適化（`strategy/kelly.py`）で、複勝（`TicketType.PLACE`）が `_ticket_hits` に未対応。

### チェックリスト

- [ ] 複勝を `allowed_ticket_types` に含めていないか（含めると正しく動作しない）

**対策**: 複勝を使う場合は Kelly 最適化をスキップし、EV ベースのフラットベットで対応。

---

## 7. 場・選手の偏り

**問題**: 特定の場や選手にデータが偏ると、汎化性能が低下する。

### チェックリスト

- [ ] `rider_track_win_rate` のサンプル数閾値（5 走以上）が適切か
- [ ] 特定場のデータで過学習していないか

---

## 8. ticket_payouts と odds_snapshot の違い（最重要）

**問題**: `ticket_payouts` テーブルは**的中した組み合わせのみ**を格納している。ハズレの券種・組み合わせは記録されない。一方、`odds_snapshot` テーブルは**全組み合わせ（ハズレ含む）**のオッズを格納している。

### 絶対に守るべきルール

- **バックテストのオッズソースには `odds_snapshot`（`use_final_odds=False`）を使う**
- `ticket_payouts` をオッズソース（`use_final_odds=True` / `market_odds_from_payouts`）に使うと、**的中した券だけが候補になり、全ベットが的中する前提のバックテストになる**。結果として ROI が数百〜数千%に膨張する
- `ticket_payouts` は**精算（settlement）専用**。ベットの的中判定と払戻額の計算にのみ使用する

### チェックリスト

- [ ] `BacktestEngine.run()` の `use_final_odds` が `False` になっているか
- [ ] `_run_standardized_backtest()` で `use_final_odds=False` を指定しているか
- [ ] ROI が不自然に高い（全券種で100%超など）場合、オッズソースの誤りを疑う

### 各テーブルの正しい用途

| テーブル | 内容 | 用途 |
|---------|------|------|
| `odds_snapshot` | 全組み合わせのオッズ（確定後取得） | **EV計算・候補選定** |
| `ticket_payouts` | 的中組み合わせの払戻のみ | **精算（的中判定・払戻額計算）** |

---

## 9. 過剰フィルタによる候補ゼロ

**問題**: `min_confidence`, `min_expected_value`, `min_probability_by_ticket`, `min_odds`, `max_odds` を同時に厳しくしすぎると、開発セットでは見栄えの良い候補が出ても、ホールドアウトでは全券種で候補ゼロになることがある。

### チェックリスト

- [ ] 戦略探索時に `valid_candidates` 数を記録しているか
- [ ] `min_probability` の有無を比較しているか
- [ ] `min_odds` / `max_odds` を一度に強くしすぎていないか
- [ ] 交差検証を通した後にホールドアウトで候補ゼロになっていないか

**対策**:
- 2段階探索（coarse → fine）で有望領域だけを掘る
- `profit` だけでなく `bet_races` との Pareto front で選ぶ
- 候補ゼロが続く場合は、戦略側の制約追加ではなくモデル側（V011）の改善に進む

---

## 10. Phase 4 運用の注意

**問題**: 発走前のタイトなタイムラインで複数の処理を実行する必要がある。

### チェックリスト

- [ ] 10 分前: 試走タイム取得に失敗していないか
- [ ] 3 分前: オッズ取得に失敗していないか
- [ ] ネットワークエラー時のリトライが設定されているか（`tenacity`）

**対策**: スクレイパーに `scrape_max_retries`（既定 3）と `scrape_timeout_sec`（既定 30）を設定。

---

## 10. 特徴量追加の収穫逓減

V013（111特徴量）→V014（141特徴量）の実験で、追加30特徴量（ストリーク、キャリア統計、交互作用）がモデル品質をほぼ改善せず（NDCG +0.001）、バックテスト利益はむしろ悪化した。

**教訓**:
- 最初の大幅追加（59→111: 前走データ、FORM_SCORE、EWM）が最大効果
- 追加分は収穫逓減。LightGBMの `feature_fraction` が自然に不要特徴量を無視するが、候補が多すぎると有用特徴量の選択確率が下がる
- 特徴量追加より、コア特徴量（前走データ）の品質向上が重要

---

## 11. kelly パラメータの感度

V013 の感度分析で、kelly=0.05→0.10 に上げるだけで全ての conf 設定で利益消滅。

**教訓**:
- パリミュチュエル方式（控除率25%）では、モデルエッジが薄い場合に過大な kelly が破滅的
- 「利益が出ているから kelly を上げて利益を倍にしよう」は **最も危険な判断**
- kelly は常にエッジの実測値に基づいて設定すべき。バックテストで利益ゼロになる最小 kelly を確認し、その半分を使うのが安全
